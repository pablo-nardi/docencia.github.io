Normalización

El proceso de organizar una base de datos para eliminar la redundancia y prevenir errores. Un lugar para cada dato, y cada dato en su lugar.

01
¿Qué es la Normalización?
El problema que resuelve y cómo lo hace

La normalización es el proceso de organizar las tablas y columnas de una base de datos relacional siguiendo un conjunto de reglas llamadas Formas Normales (FN). El objetivo es eliminar la redundancia de datos y prevenir las anomalías que aparecen cuando se insertan, actualizan o eliminan registros.

def
Definición Normalizar es descomponer tablas grandes y desordenadas en tablas más pequeñas, bien definidas y relacionadas entre sí, de modo que cada tabla tenga una única responsabilidad y cada dato esté almacenado en un solo lugar.

¿Qué pasa si no normalizamos? Una base de datos sin normalizar tiene anomalías — comportamientos incorrectos que aparecen al trabajar con los datos:

Anomalía 1
Inserción
No podés agregar ciertos datos sin agregar otros que no tenés. Ej: no podés registrar un nuevo proveedor si todavía no tiene productos.
Anomalía 2
Actualización
Si un dato se repite en varias filas, tenés que actualizarlo en todas. Si olvidás una, la base de datos tiene información contradictoria.
Anomalía 3
Eliminación
Al borrar una fila para eliminar un dato, perdés otros datos que estaban en la misma fila. Ej: al borrar el último pedido de un cliente, desaparecen sus datos.
ejemplo
Tabla sin normalizar — el problema es visible Si guardás "nombre del cliente", "ciudad del cliente" y "teléfono del cliente" en cada fila de la tabla de pedidos, tenés el nombre repetido N veces. Si el cliente cambia de teléfono, hay que actualizar N filas. Si te olvidás de una, la base de datos tiene dos teléfonos distintos para el mismo cliente — y no sabés cuál es el correcto.
02
¿Cuándo se usa?
Las señales que indican que una tabla necesita ser normalizada

La normalización se aplica durante el diseño de la base de datos, antes de cargar datos reales. También se aplica cuando una base existente presenta problemas de inconsistencia.

Señal 1
Datos repetidos
Ves el mismo valor (nombre, ciudad, precio) repetido en muchas filas de la misma tabla. Si hay que cambiarlo, hay que actualizarlo en múltiples lugares.
Señal 2
Celdas con múltiples valores
Una columna guarda varios valores en la misma celda, separados por comas o espacios: "Ana, Luis, Marta" o "Rojo/Azul". Imposible de filtrar y consultar correctamente.
Señal 3
Una tabla hace "demasiadas cosas"
La tabla mezcla información de conceptos diferentes. Una tabla "Ventas" que también guarda datos del vendedor, del cliente y del producto en las mismas filas.
Señal 4
Muchas columnas vacías (NULL)
Muchas filas tienen NULL en la mayoría de las columnas porque solo aplican a ciertos registros. Señal de que la tabla está mezclando diferentes tipos de entidades.
siempre
Regla de oro: primero se normaliza, después se desnormaliza si es necesario La normalización es el punto de partida correcto. Luego, si el sistema crece mucho y las consultas se vuelven lentas, se puede desnormalizar intencionalmente en puntos específicos para ganar velocidad. Pero eso es una decisión consciente de un experto, no el punto de partida.
03
Las Formas Normales
Las reglas que se aplican en orden, una a la vez

Las formas normales son reglas que se aplican en orden secuencial. Para estar en 2FN, primero tenés que cumplir 1FN. Para estar en 3FN, primero tenés que cumplir 2FN. En la práctica, llegar a 3FN es el objetivo estándar en la mayoría de los sistemas.

1FN
Primera Forma Normal
Eliminar grupos repetidos y valores no atómicos.

Cada columna debe contener un valor único, no listas ni conjuntos. Cada fila debe ser única (tener PK). No puede haber columnas repetidas como telefono_1, telefono_2.
2FN
Segunda Forma Normal
Estar en 1FN + eliminar dependencias parciales.

Aplica solo cuando la PK es compuesta. Todo atributo no-clave debe depender de la PK completa, no solo de una parte de ella. Si un atributo depende de la mitad de la PK, se separa en otra tabla.
3FN
Tercera Forma Normal
Estar en 2FN + eliminar dependencias transitivas.

Ningún atributo no-clave debe depender de otro atributo no-clave. Si A → B → C (C depende de B, B depende de la PK), entonces C debe ir en su propia tabla con B como PK.
→
¿Qué significa "depender de"? Un atributo B "depende funcionalmente" de A cuando dado un valor de A, siempre hay un único valor posible de B. Ejemplo: dado un dni, siempre hay un único nombre — el nombre depende del DNI. Si el nombre dependiera también de otra cosa (como del cargo en la empresa), habría una dependencia compleja.

Ejemplo rápido — el problema de la 1FN: una celda con múltiples valores

04
Ventajas y Desventajas
Qué ganás y qué cedés al normalizar
✓ Ventajas
→
Elimina redundancia. Cada dato se guarda en un solo lugar. El nombre de un cliente existe en una sola tabla, no en cada pedido.
→
Previene inconsistencias. Si el teléfono del cliente cambia, se actualiza en un solo lugar y todos los pedidos lo reflejan automáticamente.
→
Mejor integridad de datos. Las restricciones de FK garantizan que los datos relacionados sean coherentes en todo el sistema.
→
Ahorra espacio. Sin repeticiones, la base de datos ocupa menos espacio en disco, especialmente con millones de filas.
→
Facilita el mantenimiento. Agregar nuevas entidades o modificar estructuras existentes es más simple cuando cada tabla tiene una sola responsabilidad.
✗ Desventajas
→
Consultas más complejas (JOINs). Para obtener información completa hay que unir varias tablas. Una consulta simple puede requerir 4 o 5 JOINs.
→
Menor velocidad de lectura. Los JOINs son operaciones costosas. En tablas con millones de filas, unir muchas tablas puede ser lento.
→
Más tablas para gestionar. Un modelo normalizado puede tener 20 o 30 tablas donde antes había 5. Más complejidad estructural.
→
Curva de aprendizaje. Entender un modelo bien normalizado requiere conocer todas las relaciones entre tablas, lo cual no es inmediato para un desarrollador nuevo.
tradeoff
La normalización prioriza la integridad por sobre la velocidad de lectura Es un intercambio consciente. En sistemas transaccionales (ventas, stock, RRHH) donde los datos cambian constantemente, la integridad es más importante. En sistemas de reportes o analytics donde solo se lee, puede convenir desnormalizar para ganar velocidad.

Dos ejemplos reales paso a paso
Del caos a 3FN — visualizando cada transformación

Sistema de Pedidos de una tienda

Imaginá que alguien diseñó la base de datos de pedidos en una sola tabla gigante, como si fuera un Excel. Así empieza el problema:

✕  Tabla original SIN normalizar — todo en un solo lugar
id_pedido fecha dni_cliente nombre_cliente ciudad_cliente id_producto nombre_producto precio_unit cantidad
1 01/03 20123456 Ana García Rosario P01 Teclado $5.000 2
1 01/03 20123456 Ana García Rosario P02 Mouse $3.500 1
2 02/03 30987654 Luis Pérez Córdoba P01 Teclado $5.000 1
3 02/03 20123456 Ana García Rosario P03 Monitor $45.000 1
problemas
¿Qué está mal? "Ana García" y "Rosario" se repiten 3 veces. "Teclado" y "$5.000" se repiten 2 veces. Si Ana se muda a Buenos Aires, hay que actualizar 3 filas. Si el precio del Teclado sube, hay que actualizar 2 filas. Si borrás el pedido 3, perdés los datos de Ana.
1FN Primera Forma Normal — ¿hay valores no atómicos?

En este caso, cada celda ya tiene un solo valor. La tabla ya cumple 1FN. El problema viene en los siguientes pasos.

ok
1FN cumplida. No hay celdas con listas ni columnas repetidas como producto_1, producto_2. Cada celda tiene un valor único. ✓
2FN Segunda Forma Normal — eliminar dependencias parciales

La PK compuesta es (id_pedido, id_producto). Pero hay columnas que NO dependen de ambas — dependen solo de una parte:

  • fecha, dni_cliente, nombre_cliente, ciudad_cliente → solo dependen de id_pedido
  • nombre_producto, precio_unit → solo dependen de id_producto
  • cantidad → sí depende de ambos (cuántas unidades de ese producto en ese pedido)

Solución: separar en 3 tablas.

✓  Resultado después de aplicar 2FN

Pedido

id_ped PK fecha dni_cli FK
1 01/03 20123456
2 02/03 30987654
3 02/03 20123456

Producto

id_prod PK nombre precio
P01 Teclado $5.000
P02 Mouse $3.500
P03 Monitor $45.000

DetallePedido

id_ped PKFK id_prod PKFK cant.
1 P01 2
1 P02 1
2 P01 1
3 P03 1

Notar que dni_cliente quedó en Pedido como FK — eso se resuelve en 3FN.

3FN Tercera Forma Normal — eliminar dependencias transitivas

En la tabla Pedido todavía tenemos un problema: ciudad_cliente no depende de id_pedido — depende de dni_cliente, que a su vez depende de id_pedido. Eso es una dependencia transitiva: id_pedido → dni_cliente → ciudad_cliente.

Solución: separar los datos del cliente en su propia tabla.

Antes — dependencia transitiva
La tabla Pedido guarda datos del cliente (nombre_cliente, ciudad_cliente) que realmente pertenecen a una entidad distinta. Esos datos se repiten en cada pedido del mismo cliente.
Después — 3FN cumplida
Los datos del cliente van en su propia tabla CLIENTE. La tabla PEDIDO solo guarda una FK al cliente. Si el cliente cambia de ciudad, se actualiza en un solo lugar.
✓  Modelo final en 3FN — 4 tablas bien definidas

Cliente

dni PK nombre ciudad
20123456 Ana García Rosario
30987654 Luis Pérez Córdoba

Pedido

id_pedido PK fecha dni_cliente FK
1 01/03 20123456
2 02/03 30987654
3 02/03 20123456
logrado
3FN alcanzada. Cada tabla tiene una única responsabilidad. Los datos no se repiten. Actualizar la ciudad de Ana requiere cambiar una sola fila en la tabla Cliente — el cambio se refleja automáticamente en todos sus pedidos.

Sistema de Préstamos de una Biblioteca

Una biblioteca registra todos sus préstamos en una sola planilla. Cada fila es un préstamo, y tiene información del libro, del autor, de la editorial y del socio:

✕  Tabla original SIN normalizar
id_prestamo isbn_libro titulo autor editorial ciudad_editorial dni_socio nombre_socio tel_socio f_prestamo f_devolucion
1 978-001 Rayuela Cortázar Sudamericana Buenos Aires 20111222 María López 11-1234 01/03 15/03
2 978-001 Rayuela Cortázar Sudamericana Buenos Aires 30222333 Pedro Ruiz 341-999 05/03 20/03
3 978-002 El Aleph Borges Emecé Buenos Aires 20111222 María López 11-1234 10/03 25/03
problemas
¿Qué está mal? "Rayuela", "Cortázar", "Sudamericana" se repiten en los préstamos 1 y 2. "María López" y su teléfono se repiten en los préstamos 1 y 3. Si Sudamericana cambia de ciudad, hay que actualizar todas las filas que la tienen. Si el teléfono de María cambia, hay que actualizarlo en múltiples filas.
1FN Primera Forma Normal — verificación

Cada celda tiene un solo valor. No hay columnas repetidas tipo autor_1, autor_2. La tabla ya cumple 1FN.

ok
1FN cumplida. Cada préstamo ocupa una sola fila con un valor por celda. ✓
2FN Segunda Forma Normal — dependencias parciales

La PK es simple (id_prestamo), entonces técnicamente no hay PK compuesta y la 2FN se cumple automáticamente — pero hay dependencias de entidades distintas mezcladas. Separamos las entidades claras:

  • titulo, autor, editorial → dependen de isbn_libro, no del préstamo
  • nombre_socio, tel_socio → dependen de dni_socio, no del préstamo
  • f_prestamo, f_devolucion, isbn_libro, dni_socio → sí dependen del préstamo
✓  Separamos las entidades identificadas

Libro

978-001 Rayuela Cortázar Sudamericana
978-002 El Aleph Borges Emecé

Socio

20111222 María López 11-1234

Prestamo

1 978-001 20111222 01/03 15/03
2 978-001 30222333 05/03 20/03
3 978-002 20111222 10/03 25/03
3FN Tercera Forma Normal — dependencia transitiva en Libro

En la tabla Libro hay una dependencia transitiva: isbn → editorial → ciudad_editorial. La ciudad de la editorial no depende del ISBN del libro — depende de la editorial. Si Sudamericana abre una sucursal en Rosario, habría que actualizar todos sus libros.

Solución: separar la editorial en su propia tabla.

✓  Modelo final en 3FN — 4 tablas

Editorial

id_ed PK nombre ciudad
1 Sudamericana Buenos Aires
2 Emecé Buenos Aires

Libro

isbn PK titulo autor id_ed FK
978-001 Rayuela Cortázar 1
978-002 El Aleph Borges 2
logrado
3FN alcanzada. Editorial, Libro, Socio y Prestamo son 4 tablas independientes. Cada una tiene un solo concepto del mundo real. No hay datos repetidos. Si Sudamericana cambia de ciudad, se actualiza en una sola fila de la tabla Editorial.

Modelos en DBML
Los dos modelos finales normalizados, listos para pegar en dbdiagram.io

Copiá cualquiera de los dos bloques y pegalo en dbdiagram.io para visualizar el modelo normalizado completo.

Ejemplo 1 — Sistema de Pedidos (3FN): 

Table Cliente {
  dni      varchar(10)   [pk]
  nombre   varchar(100)
  apellido varchar(100)
  ciudad   varchar(80)
  telefono varchar(20)
}

Table Producto {
  id_producto  varchar(10)   [pk]
  nombre       varchar(150)
  precio_unit  float
  stock        int
}

Table Pedido {
  id_pedido    int          [pk, increment]
  fecha        date
  dni_cliente  varchar(10)  [ref: > Cliente.dni]
  // Un cliente puede tener muchos pedidos (1:N)
}

Table DetallePedido {
  id_pedido   int          [ref: > Pedido.id_pedido]
  id_producto varchar(10)  [ref: > Producto.id_producto]
  cantidad    int
  precio_al_momento float  // snapshot del precio — puede cambiar

  indexes {
    (id_pedido, id_producto) [pk]  // PK compuesta
  }
}

Ejemplo 2 — Biblioteca (3FN): 

Table Editorial {
  id_editorial  int          [pk, increment]
  nombre        varchar(100)
  ciudad        varchar(80)
  pais          varchar(60)
}

Table Libro {
  isbn          varchar(20)  [pk]
  titulo        varchar(200)
  autor         varchar(150)
  anio_pub      int
  id_editorial  int          [ref: > Editorial.id_editorial]
  // Dependencia transitiva eliminada: ciudad_editorial ahora esta en Editorial
}

Table Socio {
  dni      varchar(10)   [pk]
  nombre   varchar(100)
  apellido varchar(100)
  telefono varchar(20)
  email    varchar(120)
}

Table Prestamo {
  id_prestamo    int          [pk, increment]
  isbn_libro     varchar(20)  [ref: > Libro.isbn]
  dni_socio      varchar(10)  [ref: > Socio.dni]
  fecha_prestamo date
  fecha_devol    date
  estado         varchar(20)  // 'activo', 'devuelto', 'vencido'
}
consejo
Cómo reconocer que llegaste a 3FN Hacete estas tres preguntas para cada tabla: ¿Cada columna tiene un solo valor por fila? (1FN) ¿Cada columna no-clave depende de la PK completa? (2FN) ¿Ninguna columna no-clave depende de otra columna no-clave? (3FN) Si las tres respuestas son "sí", la tabla está en 3FN.
