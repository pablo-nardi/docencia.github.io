Desnormalización

El arte de romper las reglas de normalización a propósito, con un objetivo claro: velocidad de lectura o integridad histórica. No es el punto de partida — es una herramienta de expertos.

01
¿Qué es la Desnormalización?
El proceso inverso — y por qué tiene sentido hacerlo

La desnormalización es el proceso de introducir redundancia controlada en un modelo de base de datos que ya fue normalizado, con el objetivo de mejorar el rendimiento de las consultas de lectura o preservar datos históricos.

def
Definición Desnormalizar es tomar un modelo en 3FN (o superior) y, de manera deliberada y documentada, combinar tablas, agregar columnas redundantes o guardar datos calculados — aceptando la redundancia a cambio de velocidad de acceso o de integridad histórica.

La desnormalización NO es:

  • Un error de diseño. Si se hace correctamente es una decisión técnica consciente, no un descuido.
  • El punto de partida. Siempre se normaliza primero y luego se evalúa si desnormalizar aporta valor.
  • Lo mismo que una base mal diseñada. Una tabla caótica sin estructura no es desnormalización — es ausencia de diseño.
📐 Modelo Normalizado (3FN)
Datos en tablas separadas. Sin redundancia. Actualizar un dato cambia un solo lugar. Las consultas requieren múltiples JOIN. Ideal para sistemas transaccionales donde los datos cambian constantemente.
⚡ Modelo Desnormalizado
Datos repetidos o combinados intencionalmente. Las consultas son más rápidas porque los datos ya están juntos. Actualizar un dato puede requerir cambios en múltiples lugares. Ideal para lectura intensiva o reportes.
analogía
El índice de un libro Un libro bien organizado tiene su contenido en los capítulos (normalizado). El índice al final repite los títulos y agrega los números de página (redundancia). Esa redundancia no es un error — es una herramienta de acceso rápido. La desnormalización funciona igual: agregás redundancia para hacer más rápido el acceso.
02
¿Cuándo se usa?
Las condiciones que justifican romper la normalización

La desnormalización se aplica cuando el costo de la redundancia es menor que el costo del problema que resuelve. No es una decisión ligera — requiere medir antes de actuar.

✓  Usá desnormalización cuando...
Las consultas de lectura son lentas. Múltiples JOINs en tablas con millones de filas causan tiempos de respuesta inaceptables para el usuario.
El sistema es principalmente de lectura. Plataformas de reportes, dashboards, analytics — donde se lee muchísimo y se escribe poco.
Necesitás preservar datos históricos. El precio de un producto al momento de la venta, el nombre del cliente al momento del contrato — datos que no deben cambiar aunque la fuente original cambie.
Los datos ya están normalizados. Desnormalizás sobre una base sólida, con conciencia de qué datos se duplican y por qué.
✗  No uses desnormalización cuando...
El sistema aún no está normalizado. No tiene sentido desnormalizar una base que nunca fue normalizada — primero normalizás.
No mediste el problema. Si no tenés datos de que las consultas son lentas, no tenés justificación para agregar complejidad.
Los datos cambian muy frecuentemente. Si la fuente del dato cambia constantemente, mantener las copias sincronizadas se vuelve un problema mayor que el JOINs.
El equipo no está de acuerdo. La desnormalización requiere documentación clara y consenso — si nadie sabe por qué está un campo duplicado, se convierte en deuda técnica.
proceso
El orden correcto siempre es: normalizar → medir → desnormalizar si es necesario Primero diseñás correctamente (3FN). Después ponés el sistema en producción y medís el rendimiento real. Si detectás cuellos de botella específicos, aplicás desnormalización quirúrgica en esos puntos. No en toda la base — solo donde el dato de performance lo justifica.
03
Técnicas de Desnormalización
Las formas concretas de introducir redundancia controlada

Hay varias formas de desnormalizar. Cada una resuelve un problema distinto y tiene sus propios riesgos. No se usan todas juntas — se elige la técnica adecuada para el problema específico.

Técnica 1
Columnas redundantes
Copiar una columna de otra tabla para evitar un JOIN. Ej: guardar nombre_categoria directamente en la tabla de Producto, aunque Categoría ya existe como tabla separada.
Técnica 2
Columnas calculadas (precalculadas)
Guardar el resultado de un cálculo para no repetirlo en cada consulta. Ej: total_items_pedido en la tabla Pedido, aunque podría calcularse sumando el DetallePedido.
Técnica 3
Snapshot histórico
Guardar el valor de un dato en el momento en que ocurrió un evento, aunque ese dato cambie después. Ej: precio_al_comprar en DetallePedido — el precio del producto puede cambiar mañana, pero el precio de ese pedido ya está registrado.
Técnica 4
Tablas de resumen (summary tables)
Crear tablas que almacenan resultados agregados que se consultan frecuentemente. Ej: VentasMensuales(año, mes, total) calculada y guardada en lugar de calcularse en tiempo real con millones de filas.
Técnica 5
Combinación de tablas (merge)
Unir dos tablas que siempre se consultan juntas en una sola. Ej: fusionar la tabla Usuario y Perfil_Usuario si siempre se leen juntas y la relación es 1:1 estricta.
!
Toda desnormalización debe estar documentada Cuando agregás una columna redundante o una tabla de resumen, el código o el modelo debe dejar en claro que ese dato se repite intencionalmente, de dónde viene y qué proceso lo mantiene actualizado. Sin documentación, el equipo futuro no sabrá si es un error o una decisión.
04
Ventajas y Desventajas
Qué ganás y qué cedés al desnormalizar
✓ Ventajas
→
Consultas más rápidas. Al tener los datos ya combinados o precalculados, las consultas de lectura son notablemente más veloces — especialmente en tablas con millones de filas.
→
Menos JOINs. Una consulta que requería 5 JOINs puede hacerse en 1 o ninguno. Más simple para el motor y para el desarrollador que escribe la consulta.
→
Integridad histórica. El snapshot garantiza que los reportes del pasado sean correctos aunque los datos originales hayan cambiado. Un recibo de 2020 siempre muestra el precio de 2020.
→
Dashboards en tiempo real. Las tablas de resumen permiten que un dashboard muestre métricas complejas de forma instantánea, sin esperar que el motor calcule millones de filas.
→
Menor carga en el servidor. Precalcular y cachear resultados en la base de datos reduce la carga de CPU en los momentos de mayor tráfico.
✗ Desventajas
→
Datos duplicados. La misma información existe en múltiples lugares. Si un dato cambia, hay que actualizarlo en todos sus duplicados o la base queda inconsistente.
→
Escrituras más complejas. Una inserción o actualización que antes tocaba una tabla ahora puede tocar tres. La lógica de escritura se vuelve más compleja y propensa a errores.
→
Mayor uso de espacio. Los datos duplicados ocupan más espacio en disco. Con tablas de resumen, el volumen total de la base crece.
→
Mayor costo de mantenimiento. El equipo debe saber qué datos están duplicados, por qué, y cómo mantenerlos sincronizados. Sin documentación esto se convierte en deuda técnica.
→
Riesgo de inconsistencia. Si el proceso que actualiza los duplicados falla a mitad de camino, la base queda con datos contradictorios. Requiere transacciones o jobs de sincronización robustos.

Dos ejemplos reales revisados
Casos auténticos donde la desnormalización es la solución correcta
criterio
¿Cómo se validaron estos ejemplos? Para cada uno se verificó que: (1) el modelo normalizado tiene un problema real que no se puede resolver con índices o vistas, (2) la desnormalización resuelve ese problema, y (3) hay un costo claro de redundancia que se acepta conscientemente.

El problema del precio histórico en un sistema de ventas

Una tienda online guarda sus pedidos con referencia al producto. El modelo normalizado es impecable: el precio vive en la tabla Producto y DetallePedido solo guarda una FK al producto. Pero hay un problema grave escondido:

El fallo crítico del modelo normalizado en este caso El pedido 100 fue hecho cuando el Teclado costaba $6.000. Hoy el precio en la tabla Producto es $8.000 (hubo un aumento). Si alguien consulta el historial del pedido 100 con este modelo, verá $8.000 — que es incorrecto. El cliente pagó $6.000. El sistema está mintiendo sobre su propia historia.
Pedido 100

Producto.precio = $8.000 ❌
Pero debería ser $6.000
DESPUÉS Desnormalización — snapshot del precio al momento de compra

La solución es agregar precio_al_comprar directamente en DetallePedido. Este campo captura el precio en el momento exacto en que se realizó el pedido y nunca se actualiza — es un registro histórico.

resultado
El precio histórico ahora es correcto — para siempre El pedido 100 siempre mostrará $6.000, sin importar cuántas veces cambie el precio del Teclado en la tabla Producto. La redundancia es aceptada a cambio de integridad histórica garantizada. Este patrón se usa en todos los sistemas de e-commerce del mundo.
importante
¿Por qué este caso NO se puede resolver normalizando mejor? Algunos podrían pensar en crear una tabla HistorialPrecios(id_producto, precio, fecha_desde, fecha_hasta). Eso funciona pero agrega complejidad: la consulta del pedido requeriría un JOIN extra con condición de rango de fechas. El snapshot es más simple, más rápido y semánticamente correcto: "este fue el precio en ese momento".

Dashboard de métricas de ventas — el problema del tiempo real

Una plataforma de e-commerce tiene un dashboard interno que el equipo comercial consulta todo el día. Muestra el total de ventas por mes, la cantidad de pedidos, y el ticket promedio. El modelo está correctamente normalizado en 3FN.

PROBLEMA Consultar el dashboard es lento — el modelo normalizado no alcanza

Para mostrar "Ventas de Febrero", la consulta normalizada necesita hacer:

SELECT
  YEAR(p.fecha) AS anio,
  MONTH(p.fecha) AS mes,
  SUM(dp.cantidad * dp.precio_unit) AS total_ventas,
  COUNT(DISTINCT p.id_pedido) AS cantidad_pedidos,
  AVG(sub.total_pedido) AS ticket_promedio
FROM Pedido p
  JOIN DetallePedido dp ON p.id_pedido = dp.id_pedido
  JOIN (
    SELECT id_pedido, SUM(cantidad * precio_unit) AS total_pedido
    FROM DetallePedido GROUP BY id_pedido
  ) sub ON p.id_pedido = sub.id_pedido
GROUP BY YEAR(p.fecha), MONTH(p.fecha)
ORDER BY anio, mes;
-- Con 10 millones de filas en DetallePedido: 8-15 segundos de respuesta ❌
problema
El dashboard tarda 8 a 15 segundos en cargar Con millones de pedidos históricos, esa consulta recorre todas las filas cada vez que alguien abre el dashboard. Si el equipo comercial lo abre 50 veces por día, son 50 operaciones pesadas en la base de datos. El sistema se vuelve lento para todos.
SOLUCIÓN Tabla de resumen — desnormalización para lectura veloz

La solución es crear una tabla Resumen_Ventas_Mensual que precalcula y guarda los resultados. Un proceso automático la actualiza cada hora o al cierre de cada día.

Nueva tabla: Resumen_Ventas_Mensual
id PK aniomes total_ventas calculado cant_pedidos calculado ticket_promedio calculado ultima_actualizacion
1202601$4.820.000312$15.44901/02 23:59
2202602$5.130.000341$15.04428/02 23:59
3202603$2.140.000158$13.54415/03 08:00

La consulta del dashboard ahora es trivial:

-- Con la tabla de resumen: consulta instantánea ✓
SELECT anio, mes, total_ventas, cant_pedidos, ticket_promedio
FROM Resumen_Ventas_Mensual
ORDER BY anio, mes;
-- Tiempo de respuesta: < 10 ms ✓
trade-off
El costo aceptado Los datos de la tabla de resumen pueden tener hasta 1 hora de retraso (o lo que el proceso de actualización tarde). Si alguien hace un pedido ahora, el total del dashboard no lo refleja de inmediato — lo hará en la próxima actualización. Para un dashboard gerencial, ese retraso es aceptable. Para un sistema de stock en tiempo real, no lo sería.
→
¿Los datos originales siguen normalizados? Sí. Las tablas Pedido, DetallePedido, Cliente y Producto siguen en 3FN y sin cambios. La tabla de resumen es adicional — no reemplaza las tablas normalizadas, las complementa. Los reportes detallados siguen usando las tablas originales; solo el dashboard usa la tabla de resumen.

Modelos en DBML
Los dos modelos desnormalizados, listos para pegar en dbdiagram.io

Copiá cualquiera de los dos bloques y pegalo en dbdiagram.io para visualizar el modelo completo.

Ejemplo 1 — E-commerce con snapshot de precio:

Table Cliente {
  dni      varchar(10)   [pk]
  nombre   varchar(100)
  apellido varchar(100)
  email    varchar(120)
}

Table Producto {
  id_producto   varchar(10)   [pk]
  nombre        varchar(150)
  precio_actual float         // precio vigente hoy
  stock         int
}

Table Pedido {
  id_pedido   int          [pk, increment]
  fecha       datetime
  dni_cliente varchar(10)  [ref: > Cliente.dni]
  estado      varchar(20)  // 'pendiente', 'enviado', 'entregado'
}

Table DetallePedido {
  id_pedido        int          [ref: > Pedido.id_pedido]
  id_producto      varchar(10)  [ref: > Producto.id_producto]
  cantidad         int
  precio_al_comprar float       // DESNORMALIZADO — snapshot historico
  // Este campo copia Producto.precio_actual al momento de crear el pedido.
  // Nunca se actualiza. Garantiza que los reportes historicos sean correctos
  // aunque el precio del producto cambie en el futuro.

  indexes {
    (id_pedido, id_producto) [pk]
  }
}

Ejemplo 2 — Dashboard con tabla de resumen:

// ── Modelo normalizado (sin cambios) ───────────────────
Table Pedido {
  id_pedido   int      [pk, increment]
  fecha       datetime
  dni_cliente varchar(10)
}

Table DetallePedido {
  id_pedido   int         [ref: > Pedido.id_pedido]
  id_producto varchar(10)
  cantidad    int
  precio_unit float

  indexes { (id_pedido, id_producto) [pk] }
}

// ── Tabla de resumen DESNORMALIZADA ────────────────────
Table Resumen_Ventas_Mensual {
  id                  int       [pk, increment]
  anio                int
  mes                 int
  total_ventas        float     // DESNORMALIZADO — precalculado
  cantidad_pedidos    int       // DESNORMALIZADO — precalculado
  ticket_promedio     float     // DESNORMALIZADO — precalculado
  ultima_actualizacion datetime // cuando fue recalculado por ultima vez

  indexes { (anio, mes) [unique] }
  // Esta tabla NO reemplaza las tablas normalizadas.
  // Es una copia calculada para lectura rapida del dashboard.
  // Se actualiza via job automatico cada hora o al cierre del dia.
  // Para reportes detallados usar Pedido + DetallePedido directamente.
}
resumen
Los dos patrones de desnormalización más comunes El snapshot histórico (precio al comprar) resuelve el problema de integridad temporal — los datos del pasado no deben cambiar aunque la fuente original cambie. La tabla de resumen resuelve el problema de velocidad de lectura — las métricas ya están calculadas y no se recalculan en cada consulta. Son herramientas distintas para problemas distintos.
