01
¿Cuándo se usa una relación recursiva?
La señal que indica que una entidad se relaciona consigo misma

Una relación recursiva (también llamada reflexiva o auto-relación) ocurre cuando las dos entidades que participan en la relación son la misma. En el diagrama, la línea de relación sale de una tabla y vuelve a conectar con la misma tabla.

def
Definición Una relación recursiva es aquella en la que una entidad A se relaciona con instancias de sí misma. Los registros de la tabla se vinculan entre sí sin involucrar otra tabla.

¿Cuándo aparece en la práctica? La señal más clara es cuando al analizar el dominio encontrás que dos cosas del mismo tipo se relacionan entre sí:

Señal 1
Jerarquías del mismo tipo
Un elemento del dominio puede ser "padre" o "jefe" de otro elemento del mismo tipo. Jefes que supervisan empleados (todos son empleados). Categorías que contienen subcategorías.
Señal 2
Composición del mismo tipo
Un elemento puede estar compuesto por otros elementos del mismo tipo. Un producto formado por subproductos. Un documento que incluye subdocumentos.
Señal 3
Redes y conexiones
Elementos del mismo tipo que se conectan entre sí. Personas que se siguen en una red social. Ciudades conectadas por rutas (todos son el mismo tipo).
!
¿Cómo saber que realmente necesitás recursiva? La pregunta clave es: ¿las dos partes de la relación son del mismo tipo de entidad? Si la respuesta es "sí" — por ejemplo, un Empleado es supervisado por otro Empleado — entonces necesitás recursiva. Si las dos partes son tipos distintos (Empleado supervisado por Gerente, siendo Gerente una entidad diferente), no es recursiva.
→
Las dos variantes Una relación recursiva puede tener cardinalidad 1 a N (un jefe tiene muchos subordinados, cada subordinado tiene un solo jefe) o N a N (un producto se compone de muchos otros productos, y un mismo subproducto puede entrar en muchos productos). El tratamiento en la base de datos es diferente para cada caso.
02
¿Dónde va la Clave Foránea?
La FK en una recursiva apunta a la propia tabla

Lo que hace especial a la relación recursiva es que la FK referencia a la PK de la misma tabla. La tabla es a la vez la "tabla padre" y la "tabla hijo".

Recursiva 1:N — FK en la misma tabla
La FK se agrega como una columna más dentro de la misma tabla. En el caso de Empleados: se agrega la columna id_jefe que apunta a Empleado.id_empleado. Un empleado sin jefe tiene ese campo en NULL (es el nivel más alto).
Recursiva N:N — tabla intermedia
Igual que en un N:N normal, se necesita una tabla intermedia. La diferencia es que ambas FK de la tabla intermedia apuntan a la misma tabla. En Productos: la tabla Composicion tiene id_producto e id_componente, ambas FK → Producto.
NULL
El NULL tiene un significado especial en recursivas 1:N En la FK recursiva, el valor NULL no es un error — significa "este registro no tiene padre" o "es el nodo raíz de la jerarquía". En el ejemplo del organigrama: el Director General tiene id_jefe = NULL porque no tiene superior.
→
¿Puede una FK apuntar a la misma tabla? Sí, completamente válido en SQL. Se llama self-referencing foreign key. El motor de base de datos la acepta sin problemas. La única consideración es que al cargar el "padre" antes que el "hijo" para respetar la integridad referencial, o desactivarla temporalmente en inserciones masivas.

Ejemplo Recursiva 1 a N
Organigrama de Empleados — un jefe supervisa a muchos, cada uno tiene un solo jefe

El ejemplo clásico de recursiva 1:N es un organigrama: un empleado puede ser jefe de muchos otros empleados, pero cada empleado tiene como máximo un solo jefe directo. Todos son empleados — no hay una tabla separada "Jefes".

regla
Regla de negocio "Un empleado puede supervisar a cero o muchos empleados. Cada empleado tiene como máximo un jefe directo (o ninguno, si es el nivel más alto)."

¿Por qué no usar dos tablas (EMPLEADO y JEFE)? Porque un jefe también es empleado — tiene su propio contrato, salario y puede a su vez tener un jefe. Si usarás una tabla separada para jefes, necesitarías duplicar todos los datos del empleado en ella, y cuando alguien asciende tendrías que migrar su fila. La recursiva evita todo eso.

Composición de Productos — un producto contiene otros productos

criterio
¿Cómo se validaron? Para cada ejemplo se verificó que: (1) las dos partes de la relación sean del mismo tipo de entidad, y (2) que usar dos tablas separadas crearía redundancia o perdería capacidad de modelar correctamente la jerarquía. Los dos ejemplos solo pueden modelarse con recursiva.

Organigrama de empleados en una empresa

En el sistema de RRHH de una empresa, cada empleado puede supervisar a otros empleados. El jefe directo de un empleado también es un empleado del sistema — tiene su propio legajo, salario, área y también puede tener su propio jefe.

¿Por qué es recursiva?
El jefe es un empleado
Un Gerente es un Empleado. Un Director es un Empleado. El CEO es un Empleado. No existe un tipo separado "Jefe" — todos pertenecen a la misma entidad. La relación conecta empleados con empleados.
¿Por qué NO se puede sin recursiva?
La jerarquía tiene profundidad variable
Una empresa puede tener 2 niveles o 10 niveles jerárquicos. Si usaras columnas separadas (id_supervisor_nivel1, id_supervisor_nivel2...) no sabrías de antemano cuántas columnas crear. La recursiva permite N niveles con una sola FK.

Categorías y subcategorías en un E-Commerce

En una tienda online, las categorías pueden estar anidadas dentro de otras categorías, y una misma subcategoría puede pertenecer a más de una categoría padre. Por ejemplo, "Auriculares Bluetooth" puede estar bajo "Electrónica" y también bajo "Accesorios para Celular" — ambas son categorías del mismo tipo.

¿Por qué es recursiva?
Una categoría contiene categorías
Tanto "Electrónica" como "Auriculares Bluetooth" son CATEGORIA. La relación es entre categorías — no entre dos entidades distintas. Una categoría padre y una categoría hija son del mismo tipo.
¿Por qué NO se puede sin recursiva?
El anidamiento no tiene límite
Electrónica → Celulares → Accesorios → Auriculares → Inalámbricos. Pueden ser N niveles y una subcategoría puede tener múltiples padres. Imposible modelarlo con columnas fijas o tablas separadas por nivel.
N:N
¿Por qué es N:N y no 1:N? Porque una subcategoría puede pertenecer a más de una categoría padre. "Auriculares Bluetooth" pertenece tanto a "Electrónica" como a "Accesorios para Celular". Si fuera 1:N, cada subcategoría solo podría tener una categoría padre.

Modelos en DBML
Listos para pegar en dbdiagram.io

Copiá cualquiera de los dos bloques y pegalo en dbdiagram.io para visualizar el diagrama automáticamente.

Caso 1 — Organigrama de Empleados (recursiva 1:N):

Table Empleado {
  id_empleado  int          [pk, increment]
  nombre       varchar(100)
  apellido     varchar(100)
  cargo        varchar(80)
  area         varchar(80)
  salario      float
  id_jefe      int          [ref: > Empleado.id_empleado]
  // id_jefe puede ser NULL (empleado sin jefe = nivel raiz)
  // ref: > indica relacion 1:N (el jefe tiene muchos subordinados)
  // La linea de relacion vuelve a la misma tabla = recursiva
}

Caso 2 — Categorías y Subcategorías de E-Commerce (recursiva N:N):

Table Categoria {
  id_categoria  int          [pk, increment]
  nombre        varchar(120)
  descripcion   text
  activa        boolean
}

Table Cat_Padre_Hijo {
  id_padre  int  [ref: > Categoria.id_categoria]
  id_hijo   int  [ref: > Categoria.id_categoria]

  indexes {
    (id_padre, id_hijo) [pk]  // PK compuesta
  }
  // Ambas FK apuntan a la misma tabla = recursiva N:N
  // Una subcategoria puede tener multiples padres
}
nota
Nota sobre dbdiagram.io y recursivas En dbdiagram.io, cuando la FK referencia a la misma tabla, el diagrama visual muestra una flecha que sale y vuelve a la misma caja — esa es la representación gráfica de la recursiva. Para el caso N:N, la tabla pivote muestra dos flechas apuntando a la misma entidad.
