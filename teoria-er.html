01
Archivo y Base de Datos
Conceptos fundamentales de almacenamiento
Concepto base
Archivo
Un conjunto de bits almacenados en un dispositivo (disco duro, SSD). Como una nota en el Bloc de Notas: una bolsa digital donde tirás datos (texto, números, imágenes).
Concepto principal
Base de Datos (BD)
Una colección organizada de datos estructurados, gestionada por un DBMS que permite acceso rápido, seguridad y relaciones entre tablas.
Tipo específico
BD Relacional
Datos organizados en tablas (filas y columnas) relacionadas entre sí. Como varias hojas de Excel conectadas: Clientes, Mascotas, Turnos.
?
¿Una BD es un archivo? Técnicamente sí: los datos terminan en archivos físicos en el disco. La diferencia es que están encriptados y gestionados por el DBMS. Si abrís el archivo con un editor de texto, ves símbolos que no se entienden.

¿Por qué los archivos de una BD parecen encriptados? Por tres razones que se desarrollarán en detalle más adelante:

  • Velocidad: usa índices internos para encontrar datos entre millones de registros en milisegundos.
  • Seguridad: permisos por usuario y datos encriptados. Si alguien roba el archivo físico, no puede leerlo.
  • Integridad: el motor "vigila" que lo que se guarda sea correcto, completo y consistente.
02
Modelo Entidad-Relación
Los tres pilares del diseño lógico

Un modelo de datos es un sistema formal que describe los datos según reglas predefinidas, representando los aspectos del mundo real relevantes para el sistema. El modelo que se usa es el ER (Entidad-Relación).

Pilar 1 — El "Qué"
Entidad
Objeto o concepto del mundo real sobre el que se desean guardar datos. Se dibuja como un rectángulo con el nombre en mayúsculas.

Ej: CLIENTES, MASCOTAS, TURNOS.
Pilar 2 — El "Vínculo"
Relación
La asociación entre dos entidades, definida por un verbo.

Ej: un Cliente tiene una Mascota. Un Alumno cursa una Materia.
Pilar 3 — Las Propiedades
Atributo
Las propiedades o características que describen a una entidad.

Ej: un Cliente tiene nombre, DNI, teléfono, localidad.

03
Restricciones (CONSTRAINT)
Reglas que el motor aplica para garantizar la calidad de los datos
2.2.1
NULL / NOT NULL
Define si un atributo puede quedarse vacío. El piso de un domicilio puede ser NULL. La Razón Social nunca puede serlo.
2.2.2
UNIQUE
Garantiza que no se repitan valores en una columna. Puede haber varias restricciones UNIQUE en la misma tabla. Puede permitir NULL.
2.2.3
Primary Key (PK)
Identificador único de cada registro. Tiene implícitas las restricciones NOT NULL + UNIQUE. Puede ser compuesta (varias columnas juntas).

Ejemplo — tabla Mascotas:

Bien implementado
Usar id_mascota (entero autoincremental) como PK. Es único e irrepetible. Dos mascotas pueden llamarse "Fido" sin problema.
Mal implementado
Usar el nombre de la mascota como PK. Si dos mascotas se llaman "Fido", la base de datos rechaza el segundo registro.

Con el id_mascota como PK, dos mascotas pueden tener el mismo nombre sin conflicto.

04
Cardinalidad y Clave Foránea
Cuántas instancias de una entidad se relacionan con cuántas de otra
Tipo
1 : 1
Un registro de A ↔ exactamente un registro de B.
Ej: una Persona tiene un solo Pasaporte.
Tipo — el más usado
1 : N
Un registro de A ↔ muchos de B. Un B ↔ solo un A.
Ej: un Cliente tiene muchas Mascotas.
Tipo
N : N
Muchos de A ↔ muchos de B.
Ej: un Alumno cursa muchas Materias y viceversa.
!
Reglas de Negocio Para determinar la cardinalidad correcta hay que entender cómo la organización necesita guardar sus datos. Ej: "Un Cliente puede tener una o muchas Mascotas. Una Mascota solo pertenece a un Cliente." → Relación 1:N.

Clave Foránea (FK) — Es el "puente". Una columna en la tabla del lado N que referencia la PK de la tabla del lado 1. Relaciona información sin repetirla.

Correcto
La tabla Mascotas tiene dni_dueño que apunta a Clientes.dni. El dato del cliente se guarda una sola vez; las mascotas lo referencian.
Error común
Repetir todos los datos del Cliente dentro de cada fila de Mascotas. Si el teléfono cambia, hay que actualizarlo en cada mascota — riesgo de inconsistencia.
id_mascota PKnombretipodni_dueño FK
1FidoPerro20123456
2MichiGato20123456
3RexPerro30987654

Fido y Michi pertenecen al mismo dueño (dni 20123456) sin duplicar ningún dato del cliente.

05
Cardinalidad N a N
Muchos a Muchos — por qué siempre necesita una tabla intermedia

Una relación N a N ocurre cuando múltiples registros de A se relacionan con múltiples de B, y viceversa.

→
Ejemplo — UniversidadRegla de negocio: "Un Alumno cursa muchas Materias. En una Materia pueden inscribirse varios Alumnos."
Planteamiento incorrecto
Poner FK en Alumnos → Materias y otra FK en Materias → Alumnos. Esto impide que un alumno tenga más de una materia: la columna admite un solo valor por fila.
Solución correcta
Crear una tabla intermedia (ej: Inscripciones). Alumnos y Materias tienen cada una una relación 1:N con ella. El N:N desaparece.

06
Velocidad, Seguridad e Integridad
Por qué una BD es mucho más que un archivo de texto
A
Velocidad — ÍndicesUn .txt se lee línea por línea. Con 1 millón de libros, buscar "El Quijote" es lentísimo. Las BD usan índices (como el índice de un libro físico) para saber exactamente dónde está el dato. Resultado: milisegundos.
B
Seguridad — Control de AccesoCon un .txt, cualquiera puede modificarlo. Una BD permite definir permisos por usuario: el recepcionista ve solo nombres, sin acceso a deudas ni historia clínica. Los datos viajan encriptados.

Integridad (Calidad de la Información) — El concepto más importante. Garantiza que la información sea correcta, completa y confiable. Se divide en tres reglas:

I
Integridad de Entidad — Identidad únicaCada registro debe ser único. La PK nunca puede estar vacía ni repetirse. Ej: no pueden existir dos clientes con el mismo DNI. La base de datos frena el intento.
II
Integridad Referencial — Puentes sanosLas conexiones entre tablas deben existir de verdad. No podés referenciar algo que no existe. Regla de oro: no podés borrar un Cliente si tiene Mascotas. Eso las dejaría "huérfanas", rompiendo la lógica.
III
Integridad de Dominio — Filtro de calidadEl tipo de dato que entra debe ser el correcto para ese campo. Ej: si Talle solo acepta 'S', 'M' o 'L', no podés escribir "Rojo" ni "500". También evita edades negativas o fechas inexistentes.
07
Relaciones Reflexivas o Recursivas
Cuando una entidad se relaciona consigo misma

Una relación reflexiva ocurre cuando una entidad tiene una FK que apunta a su propia PK. La misma tabla actúa como "padre" e "hijo".

Caso 1:N
Jerarquía interna
Un Veterinario puede ser mentor de otro. La tabla Veterinarios tiene una columna id_mentor que apunta a su propia id_veterinario (PK).
Caso N:N
Composición
Un insumo puede componerse de otros insumos. Requiere tabla intermedia Composicion(id_padre, id_hijo). Ej: un Kit contiene Hilos, Agujas, Dedales.

Jerarquías (Especialización)
Una entidad general que se divide en tipos específicos

Una jerarquía ocurre cuando una entidad padre tiene atributos comunes a todos sus subtipos, y cada subtipo (hijo) agrega atributos propios. Evita columnas vacías innecesarias.

→
Ejemplo — BibliotecaLa entidad SOCIO se divide en: Socio Estudiante (agrega Carnet Universitario) y Socio Profesional (agrega Número de Matrícula). Ambos comparten Nombre y DNI.

09
Modelo Relacional
Representación lógica — estilo dbDiagram.io

El modelo relacional es la representación escrita de las tablas y sus relaciones. Antes de crear la BD real, se escribe este modelo para revisar la estructura y compartirla con el equipo.

Table Clientes {
  dni       int     [pk]
  nombre    varchar
  localidad varchar
}

Table Mascotas {
  id        int     [pk, increment]
  nombre    varchar
  tipo      varchar
  dni_dueño int     [ref: > Clientes.dni]   // Relación 1:N
}

Table Inscripciones {
  legajo      varchar  [ref: > Alumnos.legajo]
  id_materia  int      [ref: > Materias.id_materia]
  fecha_inscr date

  indexes {
    (legajo, id_materia) [pk]   // PK compuesta
  }
}
pk — clave primaria ref: > — FK apunta hacia increment — autoincremental indexes — define PK compuesta

Normalización y Desnormalización
Organizar datos sin redundancia — y cuándo romper esa regla

Normalización: proceso de organizar los datos para eliminar la redundancia (datos repetidos) y las anomalías. "Un lugar para cada cosa y cada cosa en su lugar."

Ventajas
Ahorra espacio en disco. Un cambio de nombre se corrige en un solo lugar. Mantiene la integridad. No hay datos inconsistentes.
Desventajas
Las consultas son más complejas (necesitan JOINs entre muchas tablas). Puede hacer el sistema un poco más lento al buscar.

Ejemplo — Biblioteca sin normalizar (el problema):

ID_LibroTituloAutorNacionalidadEditorial

2BestiarioJulio CortázarArgentinoSudamericana
3El AlephJorge L. BorgesArgentinoEmecé
!
El problemaSi Cortázar tuviera 100 libros, escribiríamos "Julio Cortázar" y "Argentino" 100 veces. Si nos equivocamos en uno, la base de datos "miente" — un registro dice una cosa y otro dice otra.

Normalizado — la solución (dos tablas separadas):

Tabla Autores

id_autor PKNombreNac.
10Julio CortázarARG
11J.L. BorgesARG

Tabla Libros

1Rayuela10
2Bestiario10

Desnormalización — el proceso inverso: repetir datos a propósito para ganar velocidad. Solo cuando el sistema ya es muy grande y la velocidad de lectura es prioritaria.

→
Ejemplo — Veterinaria de emergenciaSistema normalizado: para ver el teléfono del dueño hay que saltar Consultas → Mascotas → Clientes. En una emergencia eso tarda. La versión desnormalizada repite el teléfono directamente en Consultas.

★
Consejo clavePrimero siempre se normaliza. La desnormalización es un "truco" de expertos para cuando el sistema se pone lento. Si desnormalizás desde el principio, vas a crear un sistema difícil de mantener.
